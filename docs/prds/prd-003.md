## 1. 問題描述

本站（`recipe.bloss0m.com`）已完成 MVP 建站、80 篇食譜內容與集合頁 SEO 強化，但**使用者體驗與品牌識別**仍停留在第一階段「功能可用」狀態，尚未與 Bloss0m 生態的其他產品形成一致、可記憶的品牌印象。

主要痛點：

### 1.1 品牌識別不一致

- 站內對外名稱為「今天煮什麼」，僅在 about/contact 提及 bloss0m 主站。
- Bloss0m 生態已有固定命名模式（`bloss0m` 主站、`BloomRender` 繪圖入口、`Bloom Picker` 色票站），食譜站缺少對應的 **Bloom 系列垂直品牌名**。
- Header / Footer / SEO title 未呈現「這是 Bloss0m 生態中的哪一個產品」，跨站信任與回流弱。

### 1.2 行動版與導覽體驗不足

- 主選單在 720px 以下改為**垂直堆疊 6 個連結**（食譜、食材、情境、義大利麵、牛肉、冰箱剩料），佔用首屏、不易掃讀。
- 首頁內容區塊持續增加（情境、牛肉、義大利麵、熱門食材…），**資訊優先順序**對「今晚要煮什麼」的核心任務不夠聚焦。
- 首頁食材搜尋 → 冰箱工具需跳頁，動線可再縮短、回饋可再清楚。

### 1.3 食譜瀏覽與工具頁可讀性

- 食譜列表篩選為前端即時過濾，**無 URL 狀態**（無法分享篩選結果、重新整理後狀態消失）。
- RecipeCard 資訊密度尚可，但缺少一致的「3 秒判斷」視覺層級強化（時間 / 設備 / 命中食材）。
- 冰箱工具結果排序邏輯正確，但空結果、輸入錯誤、偏好切換的**引導文案**仍可更友善。

## 2. 證據

- `src/lib/site.ts` 的 `siteName` 仍為「今天煮什麼」，與 Bloss0m 生態其他產品命名不一致。
- `Header.astro` 在窄螢幕下 6 連結垂直排列（`global.css` `@media (max-width: 720px)`），行動使用情境佔比高時會直接影響第一印象。
- PRD-001 / UI guideline 已定義「冰箱便條感、3 秒判斷」設計方向，但第三階段尚未做**系統性 UX 收斂**。
- PRD-002 以內容量與 SEO 為主；使用者回訪、站內動線、品牌記憶尚未被當作獨立工作項處理。

## 3. 品牌命名定義（本 PRD 核心決策）

### 3.1 Bloss0m 生態現況

| 產品 | 角色 | URL | 託管 |
| --- | --- | --- | --- |
| **bloss0m** | 個人內容與實驗主站 | `https://www.bloss0m.com/` | GitHub Pages |
| **Bloom Picker** | 色票推薦 | `https://www.bloss0m.com/bloom-picker/` | GitHub Pages（主站路徑） |
| **Bloom Render** | AI / 繪圖入口 | `https://www.bloss0m.com/bloom-render/` | GitHub Pages（主站路徑） |
| **Bloom Kitchen**（本 PRD） | 台灣家常食譜 SEO | `https://recipe.bloss0m.com/` | Cloudflare Pages（獨立子網域） |

主站產品採 `bloom-{功能}` 路徑；食譜站因部署在 Cloudflare Pages，**維持獨立子網域**，不在本 PRD 遷移至主站路徑。

### 3.2 推薦品牌名：**Bloom Kitchen**

**正式名稱：** Bloom Kitchen  
**中文副標／口號：** 今天煮什麼  
**對外一句話：** 從冰箱剩料出發，快速找到今晚能煮的一人份料理。

**推薦理由：**

1. **與生態一致** — 延續 `Bloom` 前綴，和 BloomRender、Bloom Picker 同一產品家族。
2. **語意清楚** — Kitchen 明確指向「料理 / 食譜」垂直，不與繪圖、色票混淆。
3. **保留 SEO 與在地性** — 「今天煮什麼」作為中文副標與 H1/描述保留，不犧牲既有關鍵字與使用者理解。
4. **顯示層級建議** — Header 主標：`Bloom Kitchen`；副標或 slogan 小字：`今天煮什麼`；`<title>` 格式：`{頁面標題}｜Bloom Kitchen · 今天煮什麼`。

### 3.3 備選方案（未推薦）

| 方案 | 優點 | 缺點 |
| --- | --- | --- |
| B. 維持「今天煮什麼」為唯一品牌 | 零遷移成本 | 無法與 Bloom 生態對齊，跨站識別弱 |
| C. **Bloom Cook** | 簡短 | 較像動詞工具頁，垂直感不如 Kitchen |
| D. **Bloom Pot** | 有料理聯想 | 與 Bloom Picker 的「名詞+工具」節奏不完全一致，且 pot 較口語 |

### 3.4 品牌關係圖

```text
www.bloss0m.com（GitHub Pages 主站）
├── /bloom-picker/   → Bloom Picker
├── /bloom-render/   → Bloom Render
└── （外部連結）     → Bloom Kitchen @ recipe.bloss0m.com（Cloudflare Pages）
                              └── 中文口號：今天煮什麼
```

### 3.5 URL 策略（Human 已確認：方案 A）

**決策：只改品牌，不改 URL。**

| 項目 | 決策 |
| --- | --- |
| 產品品牌名 | **Bloom Kitchen** |
| 中文副標 | **今天煮什麼** |
| 正式網址 | **維持** `https://recipe.bloss0m.com/` |
| 主站路徑 | **不建立** `www.bloss0m.com/bloom-kitchen/` redirect |
| 子網域更名 | **不進行**（例如 `kitchen.bloss0m.com`） |
| canonical / sitemap | 仍以 `recipe.bloss0m.com` 為準，不因品牌更名而變更 host |

**對外說明文案（about / footer）：**

> Bloom Kitchen 是 Bloss0m 生態的食譜站，網址為 recipe.bloss0m.com。同系列產品包含 Bloom Picker 與 Bloom Render。

**Footer 生態連結（spec-010 實作）：**

| 標籤 | URL |
| --- | --- |
| bloss0m | `https://www.bloss0m.com/` |
| Bloom Picker | `https://www.bloss0m.com/bloom-picker/` |
| Bloom Render | `https://www.bloss0m.com/bloom-render/` |
| Bloom Kitchen | `https://recipe.bloss0m.com/`（本站） |

### 3.6 不在本 PRD 變更的項目

- **網域與託管** — 維持 `recipe.bloss0m.com` + Cloudflare Pages；不遷移至 GitHub Pages 或主站 `/bloom-kitchen/` 路徑。
- **bloss0m 主站 repo** — 不在主站新增 redirect 頁（方案 B 排除）；僅在本站 Footer 連回主站與 sibling 產品。
- **bloss0m 主站視覺** — 不重做主站導覽（若要在主站加 Bloom Kitchen 入口，屬主站 repo 另案）。

## 4. 預期成果

### 上線後 2 週（品牌 + 導覽）

- 全站 `siteName`、Header、Footer、about、SEO title 完成 **Bloom Kitchen** 品牌統一。
- 行動版主選單改為可收合選單或精簡入口，首屏不再被 6 連結垂直堆疊佔滿。
- Footer 出現 Bloss0m 生態連結（見 §3.5 表格；含 bloss0m、Bloom Picker、Bloom Render）。

### 上線後 1 個月（動線 + 瀏覽）

- 首頁資訊架構重排：冰箱剩料入口 > 熱門情境 > 精選食譜 > 主題專區（牛肉 / 義大利麵）。
- 食譜列表篩選支援 URL query（可分享、可返回）。
- 冰箱工具：空結果 / 錯誤輸入有明確下一步建議（改食材、看熱門食譜、清除偏好）。

### 可衡量指標（輕量，非硬性 KPI）

- 行動版跳出率（Search Console / 簡易 analytics）觀察趨勢。
- 冰箱工具頁 → 食譜詳情點擊率（若已有事件追蹤則記錄；若無則 spec 內評估是否加最小 click tracking）。
- 人工走查：3 個核心路徑（首頁找食譜、列表篩選、冰箱工具）在 375px 寬度下 **≤ 3 次點擊** 到達目標食譜。

## 5. 方案選項

| 方案 | 優點 | 缺點 | 工時 |
| --- | --- | --- | --- |
| **A. 品牌 + 導覽 + 首頁動線 + 列表 UX（推薦）** | 完整 UX 提升、與 Bloom 生態對齊 | 需 4 個 spec 分批交付 | 中 |
| B. 只做品牌更名（文案 / Header） | 最快 | 不解決行動版導覽與動線痛點 | 低 |
| C. 只做行動版導覽重構 | 直接改善最大 UX 問題 | 品牌仍不一致 | 中低 |

**推薦：方案 A**，依 spec 拆分，可先做 spec-010（品牌）再並行 011～013。

## 7. 拆解為 Feature / Spec

**Spec 清單：**

- **spec-010** Bloom Kitchen 品牌識別與站內文案統一
  - 定義 `siteName`、品牌顯示層級（Bloom Kitchen / 今天煮什麼）
  - **URL 維持** `recipe.bloss0m.com`；更新 `PUBLIC_SITE_URL` 相關文案但不改 host
  - 更新 Header、Footer、SeoHead、about/contact/legal、JSON-LD author 名稱
  - Footer 加入 Bloss0m 生態連結（§3.5 四站連結）
  - 更新 `guideline/ui/ui-guideline.md` 的專案名稱與品牌章節

- **spec-011** 導覽與行動版 UX
  - 主選單收合（hamburger 或 drawer），窄螢幕不再垂直堆 6 連結
  - 導覽項目分層：核心（食譜 / 食材 / 情境 / 冰箱剩料）vs 主題專區（牛肉 / 義大利麵）
  - 全站麵包屑 / 返回路徑一致性檢查

- **spec-012** 首頁與冰箱工具動線
  - 首頁區塊重排與 copy 收斂（核心任務優先）
  - 首頁搜尋 → 冰箱工具體驗優化（預填、錯誤提示、成功後結果錨點）
  - 空結果 / 低命中結果的引導模組

- **spec-013** 食譜瀏覽體驗強化
  - 列表篩選 URL state（category / time / tool / keyword query）
  - RecipeCard 資訊層級強化（時間、設備、核心食材視覺優先）
  - 食譜詳情頁可掃讀性微調（步驟、食材表、相關食譜區）

**Feature（BDD 使用，可選）：**

- [ ] `features/ux/mobile-navigation.feature`
- [ ] `features/ux/fridge-tool-empty-state.feature`
- [ ] `features/ux/recipe-list-filter-url.feature`

**拆解理由：**

- spec-010 先確立品牌，後續 UI 改動才有統一命名依據。
- spec-011 解決最大行動版痛點，且與品牌 Header 強相關，建議緊接 010。
- spec-012、013 可並行，但 012 偏首頁動線、013 偏列表/詳情，職責分離避免單 spec 過大。
