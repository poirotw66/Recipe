## 1. 問題描述

台灣租屋族、一人生活者與想快速下廚的人，常常只知道冰箱裡有什麼食材，卻不知道今天能煮什麼；市面上的食譜站多半內容龐雜、社群導向或功能過重，不利於用長尾 SEO 快速驗證「一人份、租屋、懶人料理」這個垂直需求是否有自然流量與廣告變現機會。

本專案第一階段要用 0 元月費的純靜態架構，建立一個可被 Google 收錄、可申請 AdSense、可觀察 Search Console 數據的食譜 SEO 站。

## 2. 證據

- `Spec.md` 已明確定義第一階段策略：Cloudflare Pages Free + Astro 靜態頁 + Markdown 食譜 + JSON 食材資料 + Recipe structured data + AdSense。
- Google 搜尋可理解食譜 structured data；食譜頁若提供時間、營養、圖片、FAQ、Breadcrumb 等資訊，有利於搜尋結果呈現。
- AdSense 申請需要基本政策頁與 `ads.txt` 等準備；第一階段規格已列出 `/about`、`/contact`、`/privacy-policy`、`/terms`、`/ads.txt`。
- Cloudflare Pages Free 適合純靜態內容站，且不使用 Pages Functions 時可避免 Workers 請求計費。
- SEO 驗證需要足夠頁面量；`Spec.md` 建議正式測 SEO 至少做到 300 篇食譜、100 個食材頁、30 個情境頁。

## 3. 預期成果

### 上線第 1 個月

- 上線頁面：400+。
- Google 收錄頁：100+。
- Search Console 開始出現曝光資料。
- AdSense 送審。
- 月成本：US$0。

### 上線第 3 個月

- 上線頁面：500+。
- Google 收錄頁：300+。
- 月自然流量：3,000 - 10,000 visits。
- AdSense 通過，或根據審核結果重新優化後再送。
- 第一批有效關鍵字：50+。

### 上線第 6 個月

- 食譜頁：1,000 篇。
- 食材頁：150+。
- 情境頁：50+。
- 月自然流量：30,000+ visits。
- 月廣告收入：NT$1,000 - NT$10,000 起測。

## 4. 方案選項

| 方案 | 優點 | 缺點 | 工時估算 |
| --- | --- | --- | --- |
| A. Astro + Cloudflare Pages Free + Markdown/JSON 靜態站 | 0 元月費、架構簡單、部署快、SEO 友善、無後端維護成本 | 內容管理需靠檔案與 Git；搜尋與工具只能做前端靜態比對 | 低 |
| B. Next.js + Vercel/Cloudflare + CMS | 生態成熟、互動功能擴充容易、CMS 編輯體驗較好 | 可能產生平台或 CMS 成本；架構與部署複雜度較高 | 中 |
| C. WordPress 或現成 CMS | 後台與內容管理完整、非工程角色容易上手 | 成本、維護、安全、效能與客製化風險較高 | 中 |

推薦：方案 A。

理由：第一階段核心不是功能完整，而是以最低成本驗證 SEO 收錄、自然流量與 AdSense 可行性。Astro 靜態站能用 Markdown/JSON 產生大量可收錄頁面，也能在沒有後端、資料庫與 AI API 的情況下完成 MVP。

## 5. 不在範圍內

第一階段明確不做：

- 會員登入。
- 收藏食譜。
- 留言。
- 評分。
- 後台 CMS。
- 資料庫。
- AI API。
- 圖片生成。
- 動態搜尋 API。
- Cloudflare Workers。
- Cloudflare D1。
- Cloudflare R2。

第一階段的圖片放在 `/public/images`，內容使用 Markdown + JSON 管理。

## 6. 拆解為 Feature / Spec

> 本 PRD 產生以下 spec（狀態、依賴、PRD 追溯等 metadata 由 `/vif-prd` Step 5 展開至 `docs/specs/specs-overview.md`，此處不重複維護）。

**Spec 清單：**

- spec-001 專案骨架與部署 — 建立 Astro 專案、Cloudflare Pages 部署、自訂網域、基礎 layout、Header/Footer 與 SEO head。
- spec-002 內容模型與食譜頁 — 建立 Markdown 食譜 content collection、食譜列表、食譜詳細頁與食譜資料欄位。
- spec-003 食材與情境頁 — 建立食材 JSON、情境 JSON、列表頁、詳細頁與關聯食譜導流。
- spec-004 SEO 與 structured data — 建立 canonical、Open Graph、Twitter Card、sitemap、robots、Recipe JSON-LD、FAQ JSON-LD、Breadcrumb JSON-LD。
- spec-005 AdSense 與政策頁 — 建立 about、contact、privacy policy、terms、ads.txt 與 AdSlot 元件。
- spec-006 冰箱剩料工具 — 建立純前端食材輸入、本地 JSON 比對、推薦排序與缺少食材提示。
- spec-007 第一批內容資料 — 建立 MVP 所需的食譜、食材、情境內容批次與命名/SEO 規則。

**拆解理由：**

依照 `Spec.md` 的里程碑與技術邊界拆解。`spec-001` 先確保可部署與基礎頁面框架；`spec-002`、`spec-003` 建立大量 SEO 頁面的內容來源與路由；`spec-004` 是搜尋收錄與 rich results 的核心；`spec-005` 對應 AdSense 申請條件；`spec-006` 是第一階段唯一工具型互動功能；`spec-007` 將內容生產與工程實作分離，方便後續分批驗收。

## 7. 依賴與影響

### 依賴

- Cloudflare Pages Free。
- Cloudflare 自訂網域與 DNS。
- Astro。
- Node.js 20 或 22。
- Google Search Console。
- Google AdSense。
- 食譜內容、食材資料、情境分類資料。

### Spec 依賴建議

- `spec-001` 無前置依賴，是所有後續工作的基礎。
- `spec-002` 依賴 `spec-001`。
- `spec-003` 依賴 `spec-001`，並與 `spec-002` 互相導流。
- `spec-004` 依賴 `spec-002` 與 `spec-003` 的頁面與資料模型。
- `spec-005` 依賴 `spec-001`，AdSlot 可在 `spec-002` 完成後插入食譜頁與列表頁。
- `spec-006` 依賴 `spec-002` 的食譜資料與 `spec-003` 的食材資料。
- `spec-007` 可與 `spec-002`、`spec-003` 並行，但正式驗收依賴內容模型穩定。

### 影響

- 建立正式站台資訊架構與 URL 結構，後續需避免任意改 slug 造成 SEO 損失。
- 建立內容資料格式後，後續食譜、食材、情境內容都需依此格式維護。
- SEO 與 structured data 的實作會影響 Google 收錄與 rich results 測試結果。
- AdSense 元件與政策頁會影響廣告審核與頁面體驗。
